Question 1
Explain the role of Vue.createApp() in this application.

Vue.createApp() is the entry point for creating a Vue 3 application instance. It accepts a root component options object (containing data(), methods, computed, etc.) and returns an application instance.

In this lab, it creates the Food Order Calculator application with all its reactive state and logic:

const app = Vue.createApp({
  // root options object
  data() { return { customerName: '', orders: [], ... }; },
  computed: { totalPrice() { ... } },
  methods:  { addOrder() { ... } }
});

After creation, app.mount('#app') binds the Vue instance to the <div id="app"> element in the HTML, making all Vue directives and interpolations in that element reactive and functional.

Without Vue.createApp(), Vue has no application instance and no reactive data — the HTML would render as plain static text.

Question 3
Explain how v-model supports the reactive data model.

v-model creates a two-way data binding between a form input element and a Vue data property. It is shorthand for binding the element's value to the data property and listening for input events to update that property.

<!-- Equivalent to: -->
<input :value="customerName" @input="customerName = $event.target.value">

<!-- Simplified using v-model: -->
<input v-model="customerName">

In this application:

  • v-model="customerName" on the text input instantly reflects any typed characters into the customerName data property, which is then displayed live in the Order Preview section.
  • v-model="selectedFood" on the <select> binds the selected food object so totalPrice (a computed property) recalculates automatically whenever the selection changes.
  • v-model.number="quantity" uses the .number modifier to cast the input string to a number automatically.

This reactivity means the developer does not need to manually query the DOM or write event handlers — Vue keeps the UI and data in sync at all times.

Question 4
Identify two Vue directives used in index.html and explain their functions.

1. v-for — Loops through an array and renders a block of HTML for each item.

<option v-for="food in foods" :key="food.name" :value="food">
  {{ food.name }} — RM{{ food.price }}
</option>

This renders one <option> tag for every object inside the foods array. The :key attribute gives each element a unique identity so Vue can efficiently update the DOM when the list changes.

2. v-if / v-else — Conditionally renders elements based on a boolean expression.

<p v-if="orders.length === 0">No order available.</p>
<table v-else ...> ... </table>

When the orders array is empty, the paragraph is rendered and the table is removed from the DOM entirely. Once an order is added, v-else activates and the table becomes visible instead. This ensures the table is only shown when there is data to display.

Question 5
Explain why totalPrice() is better written as a computed property.

A computed property is the right tool for totalPrice() for three reasons:

  • Caching: Vue caches the result of totalPrice() and only re-evaluates it when its reactive dependencies (selectedFood and quantity) change. If it were a regular method, it would re-run on every render — even unrelated ones — wasting processing time.
  • Declarative derivation: totalPrice is a value derived from other data. Computed properties express this relationship clearly. Methods are intended for actions (doing something), while computed properties are for values (calculating something).
  • Automatic reactivity: Vue automatically tracks which data properties the computed function reads. Whenever selectedFood.price or quantity changes, the UI that displays {{ totalPrice }} updates instantly without any manual event wiring.
// If written as a method — recalculates on EVERY render
getTotalPrice() { return this.selectedFood.price * this.quantity; }

// As computed — cached, reactive, efficient ✓
totalPrice() { return this.selectedFood.price * this.quantity; }
Question 6
Explain the benefit of moving from a single-file version to a separated index.html and app.js structure.

Separating the HTML structure from the JavaScript logic follows the principle of Separation of Concerns, and provides several practical benefits:

  • Readability: index.html only contains markup — it is immediately clear which elements appear on screen. app.js only contains logic — data, methods, and computed properties are easy to locate and read.
  • Maintainability: A designer can edit index.html without touching app.js, and a developer can update business logic in app.js without touching the HTML. This reduces the risk of introducing bugs across concerns.
  • Reusability: The same app.js logic could be linked to different HTML pages or templates without duplication.
  • Browser Caching: Browsers can cache app.js separately. If only the HTML changes, the cached script does not need to be re-downloaded.
  • Scalability: As the project grows, the single-file approach becomes unmanageable. The two-file structure is the first step toward a modular, component-based Vue architecture (such as Vue CLI or Vite projects).
